# 项目实施计划

## 目标与边界

项目目标是建立覆盖小学一至六年级的数学微知识点有向依赖图，并使每个公开结论可追溯、可审核、可版本化。第一阶段的交付物是一个可独立使用的 JSON 数据集和验证工具，而不是面向家长的应用。

图谱不作以下承诺：不复制或分发教材内容；不把教材章节顺序等同于学习因果；不以单次模型抽取替代教师或学科专家审核；不把“掌握”简化为一次对错。

## 成功定义

首个候选发布覆盖北师大版二年级下册出现的核心学习目标，并建立与 2022 年课程标准的映射。项目同时维护两个不能混称的发布轨道：`candidate`（可追溯、机器验证、待教育复核）与 `expert-verified`（经独立教育审核确认）。

`expert-verified v0.1` 的公开发布须同时满足：

1. 每个节点都有两个以上可观察的掌握证据、至少一个课程标准标识和来源记录。
2. 每条边有可检验的理由、来源和审核状态；`hard` 边有两名独立审核者同意。
3. 自动校验通过：标识唯一、引用完整、无自环、无重复边、图无环、清单计数匹配。
4. 至少完成三类场景验收：前置追溯、后续解锁、年级范围查询。
5. 发布说明明确覆盖范围、已知盲区、来源版本及数据许可。

## 数据模型决策

节点的最小单位是“可观察的学习目标”，例如“能用相同数位对齐的方法计算三位数加法”，而不是“第六单元”或“教材第 63 页例题”。一个目标可以关联多个教材版本、多个年级段和多个标准条目。

教材证据记录使用书目信息与页码定位，不保存扫描页、原文转录或题目全文。课程标准映射采用稳定的项目内标识 `cn-math-2022:<path>`；标准原文和标识表将在首个数据发布前以可再分发的引文方式补齐。

`hard` 表示缺少前置能力会使目标的通常学习路径失效，而不是“教师通常先讲它”。缺少充分论证时一律使用 `soft` 或不建边。每条边均是有方向的：`topicId` 依赖 `prerequisiteId`。

## 阶段与验收

| 阶段 | 产出 | 退出条件 |
| --- | --- | --- |
| 0. 规范与工具 | 本仓库、Schema、校验器、来源政策 | 结构和图校验可在本地运行 |
| 1. 课程骨架 | 2022 课标的主题-内容要求索引 | 标识规则经数学教育审核，能映射至小学段 |
| 2. 试点抽取 | 北师大二下的候选节点、教材映射、候选边 | 每项均有页码/标准来源，候选与批准数据隔离 |
| 3a. 证据交叉与候选发布准备 | 跨教材/教辅证据记录、自动化门禁、公开复核队列 | `candidate` 数据与专家确认数据明确分层；不将未确认结论表述为教育验证 |
| 4. `candidate-0.1.0` 发布 | 定版候选 JSON、manifest、变更说明、示例查询 | 满足候选发布门槛并复现校验通过；不依赖第 3b 阶段 |
| 3b. 专家校审（并行升级轨） | 节点粒度、证据、依赖强度审查记录 | 所有拟升级为 `hard` 的边双审，抽样节点满足定义；不阻塞阶段 4、5 或候选版本迭代 |
| 5. 横向扩展 | 人教版映射、其他年级、跨版本对照 | 新教材优先补充已有节点的独立证据；仅在能力确有差异时新增节点 |
| 6. 专家确认发布 | `expert-verified` 子图、审查记录与迁移说明 | 仅发布完成独立教育审核的范围；可分批进行 |
| 7. v1.0 | 1-6 年级核心图、公开贡献与版本治理 | 覆盖声明、审查容量和稳定 ID 策略成熟 |

## 近期执行顺序

1. 固化课标索引：以教育部 `教材〔2022〕2号` 的[中国政府网发布通知](https://www.gov.cn/zhengce/zhengceku/2022-04/21/content_5686535.htm)及其附件 5 为规范性来源；先核验小学段条目标识表，再开始批量映射。
2. 建立二下候选清单：以北师大 2024 版目录和逐页教材核验为依据，记录单元、页码、目标、类型、领域、候选标准映射和不确定性。
3. 进行目标归并：将同一能力的不同情境题合并，拆开概念理解、操作程序和策略选择等不同目标。
4. 建立候选边：先写理由和反例，再判定 `hard` / `soft`；教材顺序只能作为弱证据。
5. 证据交叉：将北师大版、人教版等教材及可合法引用的教辅材料映射到同一候选目标；教材间的一致性增强证据，差异触发粒度或表述复查，不直接证明依赖。
6. 候选发布：先公开可追溯、机器验证且有已知限制的 `candidate` 数据与复核队列；默认不把未独立确认的关系标为 `hard`。
7. 发布候选版本：完成自动门禁、典型查询走查、变更说明和已知限制后发布 `candidate-0.1.0`；此步骤不以人工专家审核为前置条件。
8. 并行质量升级：在获得教育资源后，完成学科专家审核和人工走查；仅通过双审的 `hard` 边和相关节点进入 `expert-verified` 发布范围。

## 角色与审核责任

| 角色 | 责任 |
| --- | --- |
| 数据维护者 | 维护 Schema、ID、版本、变更说明与自动校验 |
| 教材抽取者 | 建立候选目标与教材定位，不批准自身的关键结论 |
| 数学教育审核者 | 审核高风险节点的粒度、证据、标准映射和依赖强度 |
| 技术审核者 | 审核图结构、可复现性、许可和发布工件 |

单人维护阶段可以先发布 `candidate` 数据，但必须保留不确定性、来源与反例审查记录。进入 `expert-verified` 范围时，至少由两位不同人员覆盖“抽取”和“教育审核”角色；模型可以协助生成候选项、执行一致性检查或准备审查包，但不能作为独立审核者。

## 风险与应对

| 风险 | 应对 |
| --- | --- |
| 把教材顺序误写为依赖 | 强制填写边理由与来源；未证实的边降为 `soft` 或移除 |
| 知识点颗粒度失控 | 以学习目标和可观察证据为准，设立粒度审查样例 |
| 标准引用不稳定 | 使用项目内稳定 ID，并维护版本化标识表 |
| 版权风险 | 不入库教材扫描、OCR 全文、题目全文或未经授权材料 |
| 单一版本偏差 | 概念节点与教材映射分离，后续引入人教版等交叉核验 |
| 人工审核容量不足 | 以候选发布轨维持公开迭代；专家投入优先用于校准样本、争议节点与所有拟发布的 `hard` 边 |
| 维护不可持续 | 小批次 PR、语义化数据版本、自动校验和明确审核队列 |

## 发布节奏

建议以质量门禁而非固定日期驱动发布。发布标签必须包含证据层级，例如 `candidate-0.1.0` 或 `expert-verified-0.1.0`，不得用候选数据替代专家确认数据。每次发布采用 `MAJOR.MINOR.PATCH`：`MAJOR` 表示 ID 或模型不兼容变更，`MINOR` 表示新增覆盖范围，`PATCH` 表示更正、来源补充或措辞澄清。已发布 ID 永不复用；废弃节点保留迁移说明。

所有影响数据模型、来源策略、抽取方法、审核规则或发布范围的决策，都必须同步写入 `docs/DEVELOPMENT_LOG.md`；发布时还需在日志中记录数据版本、校验结果与已知限制。
