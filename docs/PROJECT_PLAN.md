# 项目实施计划

## 目标与边界

项目目标是建立覆盖小学一至六年级的数学微知识点有向依赖图，并使每个公开结论可追溯、可审核、可版本化。第一阶段的交付物是一个可独立使用的 JSON 数据集和验证工具，而不是面向家长的应用。

图谱不作以下承诺：不复制或分发教材内容；不把教材章节顺序等同于学习因果；不以单次模型抽取替代教师或学科专家审核；不把“掌握”简化为一次对错。

## 规范锚定原则

项目的规范锚点是教育部《义务教育数学课程标准（2022年版）》，不是任何一套教材。教材只作为私有核验样本，用于观察一个学习目标在不同版本中的呈现、表征、引入时点和粒度差异，不能决定公开图谱的节点边界或依赖强度。

公开模型分为三层：

```text
课程标准内容要求 -> 稳定能力锚点 -> 可诊断学习目标与依赖图
                                           -> 私有教材呈现证据
```

课程标准内容要求定义范围；稳定能力锚点用于跨版本比较；图谱节点表达可诊断能力；教材特有情境、例题、操作材料和算法偏好仅记录为私有证据或证据标签。不得将教材活动直接当作跨版本稳定节点。

## 成功定义

首个候选发布提供与 2022 年课程标准可定位条目相连的学习目标和依赖图。教材版本不构成公开覆盖声明；它们只提供私有验证证据。项目同时维护两个不能混称的发布轨道：`candidate`（可追溯、机器验证、待教育复核）与 `expert-verified`（经独立教育审核确认）。

`expert-verified v0.1` 的公开发布须同时满足：

1. 每个节点都有两个以上可观察的掌握证据、至少一个课程标准标识和来源记录。
2. 每条边有可检验的理由、来源和审核状态；`hard` 边有两名独立审核者同意。
3. 自动校验通过：标识唯一、引用完整、无自环、无重复边、图无环、清单计数匹配。
4. 至少完成三类场景验收：前置追溯、后续解锁、年级范围查询。
5. 发布说明明确覆盖范围、已知盲区、来源版本及数据许可。

## 数据模型决策

节点的最小单位是“可观察的学习目标”，例如“能用相同数位对齐的方法计算三位数加法”，而不是“第六单元”或“教材第 63 页例题”。每个节点必须关联一个稳定能力锚点和一个或多个可定位课程标准条目；一个锚点可关联多个图谱节点，一个节点可关联多个教材版本和年级段。

教材证据记录使用书目信息与页码定位，不保存扫描页、原文转录或题目全文。教材特定的操作、情境、算法名或章节组织优先记录为私有证据，不自动成为公开节点。课程标准映射采用稳定的项目内标识 `cn-math-2022:<path>`；标准原文不重发布，只保留官方链接、定位和独立摘要。

`hard` 表示缺少前置能力会使目标的通常学习路径失效，而不是“教师通常先讲它”。缺少充分论证时一律使用 `soft` 或不建边。每条边均是有方向的：`topicId` 依赖 `prerequisiteId`。

## 当前阶段 5.1 的实施顺序

1. 核验课程标准小学段内容要求页，建立“课程标准条目 -> 稳定能力锚点”的对照表；不得以教材目录生成锚点。
2. 审计现有 91 个节点：每项标记为“保留为可诊断节点”“合并到更稳定节点”“降为教材证据/策略标签”“暂时废弃”。
3. 为保留节点补充能力锚点 ID、课标定位、年级段和不确定性；教材来源从节点定义中移至证据层。
4. 审计 19 条 `soft` 边：确认其连接的是稳定能力或可诊断节点，而非教材活动；不升级为 `hard`。
5. 更新 Schema、校验器、迁移说明和查询样例，使能力锚点和教材证据层可机器校验。
6. 对受影响 ID 制定弃用/迁移映射，避免已发布数据静默失效。
7. 运行数据校验、发布树检查和迁移完整性检查；形成阶段 5.1 审计报告并本地提交。

## 建议下一阶段 5.2 的实施顺序

1. 以同一课标主题和能力锚点为比较范围，对五册私有教材进行跨册、跨版本检索，而不是同册对同册匹配。
2. 对每个能力锚点分别记录：跨版本支持、表述/粒度差异、教材实现差异、待定位和直接冲突。
3. 分别统计课标能力锚点重合率、可诊断节点重合率和教材实现差异率；不得混为一个“教材重合率”。
4. 对 `not-located` 进行跨册复查后才能保留；它不表示课程内容缺失。
5. 只在课程标准支持且能力边界稳定时新增候选节点；教材特有活动只进入私有证据层。
6. 运行校验、更新开发日志和跨版本报告，并在本地提交；不自动创建 Release 或远程推送。

## 角色与审核责任

| 角色 | 责任 |
| --- | --- |
| 数据维护者 | 维护 Schema、ID、版本、变更说明与自动校验 |
| 教材抽取者 | 建立候选目标与教材定位，不批准自身的关键结论 |
| 数学教育审核者 | 审核高风险节点的粒度、证据、标准映射和依赖强度 |
| 技术审核者 | 审核图结构、可复现性、许可和发布工件 |

单人维护阶段可以先发布 `candidate` 数据，但必须保留不确定性、来源与反例审查记录。进入 `expert-verified` 范围时，至少由两位不同人员覆盖“抽取”和“教育审核”角色；模型可以协助生成候选项、执行一致性检查或准备审查包，但不能作为独立审核者。

## 风险与应对

| 风险 | 应对 |
| --- | --- |
| 把教材顺序误写为依赖 | 强制填写边理由与来源；未证实的边降为 `soft` 或移除 |
| 知识点颗粒度失控 | 以学习目标和可观察证据为准，设立粒度审查样例 |
| 标准引用不稳定 | 使用项目内稳定 ID，并维护版本化标识表 |
| 版权风险 | 不入库教材扫描、OCR 全文、题目全文或未经授权材料 |
| 单一版本偏差 | 概念节点与教材映射分离，后续引入人教版等交叉核验 |
| 人工审核容量不足 | 以候选发布轨维持公开迭代；专家投入优先用于校准样本、争议节点与所有拟发布的 `hard` 边 |
| 维护不可持续 | 小批次 PR、语义化数据版本、自动校验和明确审核队列 |

## 发布节奏

建议以质量门禁而非固定日期驱动发布。发布标签必须包含证据层级，例如 `candidate-0.1.0` 或 `expert-verified-0.1.0`，不得用候选数据替代专家确认数据。每次发布采用 `MAJOR.MINOR.PATCH`：`MAJOR` 表示 ID 或模型不兼容变更，`MINOR` 表示新增覆盖范围，`PATCH` 表示更正、来源补充或措辞澄清。已发布 ID 永不复用；废弃节点保留迁移说明。

所有影响数据模型、来源策略、抽取方法、审核规则或发布范围的决策，都必须同步写入 `docs/DEVELOPMENT_LOG.md`；发布时还需在日志中记录数据版本、校验结果与已知限制。
